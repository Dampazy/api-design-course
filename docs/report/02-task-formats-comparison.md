# 2. Сравнение форматов заданий

В курс включено 6 форматов заданий, реализованных в
`backend/app/services/checker.py`. Ниже — сравнение по критериям
однозначности проверки, сложности реализации и применимости к теме API.

| Формат | Как проверяется | Плюсы | Минусы | Пример в курсе |
|---|---|---|---|---|
| **SINGLE_CHOICE** | сравнение индекса выбранного варианта | максимально однозначно, тривиальная проверка | не подходит для открытых вопросов | «Какой метод для частичного обновления?» |
| **MULTIPLE_CHOICE** | сравнение множеств индексов | проверяет более глубокое понимание (несколько верных признаков) | случайное угадывание чуть вероятнее при малом числе вариантов | «Какие методы идемпотентны?» |
| **FILL_BLANK** | нормализованное сравнение строк (регистр/пробелы) + список допустимых альтернатив | пользователь формулирует ответ сам, а не узнаёт среди вариантов | требует продумать все разумные варианты написания (alternative_answers), иначе ложные "неверно" | «Какой заголовок передаёт Bearer-токен?» |
| **ORDERING** | сравнение последовательности индексов, частичный балл по числу пар в верном относительном порядке | хорошо проверяет понимание процесса/протокола (например, OAuth2 flow) | сложнее в реализации проверки (не бинарное сравнение) | «Расставьте шаги OAuth2 Authorization Code Flow» |
| **MATCHING** | сравнение словаря пар, частичный балл по доле верных пар | проверяет знание терминологии/сопоставлений | UI сложнее (нужен выбор из списка для каждой пары) | «Сопоставьте код ответа со значением» |
| **JSON_FIX** | синтаксический разбор JSON + сравнение как структур данных (не строк) | устойчиво к форматированию/порядку ключей, реалистичная практическая задача | нужно объяснить пользователю ожидаемый формат ответа | «Приведите тело ошибки к формату RFC 7807» |
| **API_REQUEST** | парсинг свободного текста `METHOD /путь` (+ опциональные query-параметры и JSON-тело), сравнение метода/пути/query/тела по отдельности с частичным баллом | самый реалистичный формат — пользователь пишет запрос **с нуля**, а не узнаёт его среди вариантов; ближе всего к реальной работе с API | сложнее всего реализовать: нужен парсер запроса, допуск на порядок query-параметров и trailing slash | «Напишите запрос для полной замены статьи с id=15» |

## Почему не потребовался ML/эвристический разбор текста

Задания курса намеренно спроектированы так, чтобы каждый формат сводился к
**точному сравнению структурированных данных** (индексы, нормализованные
строки, JSON-объекты, последовательности, словари пар), а не к анализу
свободного текста. Это ключевое архитектурное решение: оно делает
проверку детерминированной, быстрой (без вызова внешних сервисов) и
воспроизводимой — что и было требованием задания практики
(«третий формат без ML»), реализованным сразу тремя форматами:
`ORDERING`, `MATCHING` и `API_REQUEST`.

Даже самый свободный формат — `API_REQUEST`, где пользователь пишет запрос
целиком с нуля — не требует NLP или ML: свободный текст парсится простым
детерминированным алгоритмом (первая строка — метод и путь через
`urllib.parse`, остаток — JSON-тело), а сравнение результата с эталоном —
обычное сравнение структур данных, как и во всех остальных форматах.
