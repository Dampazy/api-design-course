# 3. Архитектура решения

## Общая схема

```
┌─────────────────┐        /api/*  (Vite proxy)       ┌──────────────────┐
│   React (Vite)   │ ───────────────────────────────▶ │  FastAPI backend  │
│  localhost:5173   │ ◀─────────────────────────────── │  localhost:8000    │
└─────────────────┘             JSON                   └────────┬─────────┘
                                                                  │ SQLAlchemy ORM
                                                                  ▼
                                                          ┌───────────────┐
                                                          │  SQLite (файл) │
                                                          └───────────────┘
```

## Backend

- **FastAPI** + **SQLAlchemy 2.0** (sync ORM) + **SQLite**. Схема создаётся
  через `Base.metadata.create_all()` — без Alembic, так как схема
  фиксирована и не требует пошаговых миграций для учебного проекта.
- Три таблицы: `theory_blocks`, `tasks`, `attempts` (см.
  `backend/app/models/`). Прогресс пользователя — не отдельная таблица, а
  агрегирующий запрос по `attempts` (`services/progress_service.py`):
  это исключает рассинхронизацию данных между «прогрессом» и «попытками».
- Контент курса **не хардкожен в роутерах** — живёт в
  `backend/app/seed/theory_data.py` и `tasks_data.py`, заливается в БД
  идемпотентным скриптом `python -m app.seed.seed` (можно перезапускать
  сколько угодно раз при доработке контента).
- Проверка ответов вынесена в отдельный слой — `services/checker.py`,
  реализованный как **strategy pattern**: диспетчеризация по
  `task.task_type` на одну из 5 функций-стратегий. Ни одна стратегия не
  выбрасывает исключение наружу — `check_answer()` перехватывает все
  ошибки и возвращает `is_correct=False` с описанием причины, что и
  обеспечивает устойчивость к «провокационным примерам» (см.
  [04-edge-cases-demo.md](04-edge-cases-demo.md)).
- Аутентификация — сознательно упрощена до анонимного `session_id`
  (UUID4, генерируется на фронтенде и хранится в `localStorage`,
  передаётся в заголовке `X-Session-Id`). Это осознанное архитектурное
  решение для учебного демонстрационного проекта: даёт полноценный
  персональный прогресс и статистику без сложности полноценной регистрации
  пользователей.

## Frontend

- **React 18** + **React Router 6** + **Vite 5**. Vite proxy перенаправляет
  все запросы `/api/*` на backend, что позволяет фронтенду обращаться к
  относительным путям без хардкода `localhost:8000` (важно, например, для
  корректной работы через `docs/report/04` при демонстрации).
- Контент теории и заданий рендерится через `react-markdown` +
  `remark-gfm` (поддержка таблиц GFM) — теория курса написана в Markdown
  без привязки к конкретному frontend-фреймворку, что упрощает повторное
  использование контента (например, в `docs/student-guide/theory-conspect.md`).
- Формы ответа реализованы как 3 переиспользуемых компонента
  (`MultipleChoiceForm`, `FillBlankForm`, `OrderingForm`), покрывающих все
  6 типов заданий через параметризацию (`taskType`/`mode`), а не 6
  отдельных компонентов — избегает дублирования разметки/логики выбора.

## API (сводка)

```
GET  /api/theory                       — список тем
GET  /api/theory/{slug}                — теория конкретного блока
GET  /api/tasks?theory_block_id=&is_final_test= — список заданий с фильтрами
GET  /api/tasks/{id}                   — задание (без правильного ответа!)
POST /api/tasks/{id}/submit            — отправка и проверка ответа
GET  /api/progress                     — личный прогресс (по X-Session-Id)
GET  /api/progress/final-test          — прогресс по итоговому тесту
GET  /api/stats                        — агрегированная аналитика по всем сессиям
GET  /api/stats/tasks/{id}             — аналитика по конкретному заданию
```

Полная интерактивная спецификация (автоматически генерируется FastAPI из
аннотаций типов и Pydantic-моделей) — на `http://localhost:8000/docs`.
