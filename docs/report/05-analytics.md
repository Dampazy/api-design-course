# 5. Анализ успешности (аналитика по заданиям)

Эндпоинт `GET /api/stats` агрегирует данные из таблицы `attempts` по всем
сессиям и возвращает по каждому заданию: количество попыток
(`total_attempts`), процент верных попыток (`correct_rate`) и среднее
время ответа в мс (`avg_response_time_ms`). Реализация —
`services/progress_service.get_stats()` / `get_task_stat()` (SQL-агрегация
через `func.count`/`func.avg`, без загрузки всех строк в память).

Ниже — пример реального ответа API после нескольких демонстрационных
попыток с разных `session_id` (получено во время тестирования платформы):

| Задание | Блок | Попыток | % верных | Ср. время ответа |
|---|---|---|---|---|
| Метод для частичного обновления (SINGLE_CHOICE) | REST и HTTP-методы | 11 | 63.6% | 2.1 с |
| Соответствие кодов ответа их значению (MATCHING) | Коды ответов HTTP | 2 | 50.0% | 18.5 с |
| Шаги эволюции API без breaking changes (ORDERING) | Именование и версионирование | 2 | 50.0% | 37.5 с |

Полный ответ доступен через `GET /api/stats` или в интерфейсе на странице
«Прогресс» (`/progress`) — там же выводится персональный прогресс текущей
сессии по каждому теоретическому блоку и по итоговому тесту.

## Наблюдения (иллюстративные, на демонстрационных данных)

- Задания типа `ORDERING`/`MATCHING` показывают заметно большее среднее
  время ответа (18–37 секунд) по сравнению с `SINGLE_CHOICE` (~2 секунды)
  — ожидаемо, так как требуют многошагового взаимодействия с формой
  (расстановка порядка / выбор пары для каждого элемента), а не одного
  клика.
- Метрика `correct_rate` по отдельному заданию — потенциальный индикатор
  качества формулировки: устойчиво низкий процент верных ответов на
  конкретном задании при достаточном числе попыток может сигнализировать
  о неоднозначной формулировке, а не о сложности темы, и является поводом
  для пересмотра контента в `seed/tasks_data.py`.
- Метрика `response_time_ms` считается на фронтенде как разница между
  моментом открытия задания (`TaskDetail.jsx`, `startedAtRef`) и моментом
  отправки ответа — не учитывает время, проведённое вне вкладки браузера,
  что является известным ограничением подхода (упомянуто как direction
  for future work в [06-conclusion.md](06-conclusion.md)).
