# 4. Демонстрация работы на провокационных примерах

Требование задания практики — показать, что платформа корректно
обрабатывает некорректный/провокационный пользовательский ввод, не падая с
внутренней ошибкой (HTTP 500). Все сценарии ниже автоматизированы в
`backend/tests/test_edge_cases.py` (9 тестов, все проходят) и могут быть
воспроизведены вручную через Swagger UI (`http://localhost:8000/docs`).

| # | Сценарий | Ожидаемое поведение | Фактический результат |
|---|---|---|---|
| 1 | `POST /api/tasks/1/submit` без заголовка `X-Session-Id` | Отклонить запрос, не создавая попытку без владельца | `422 Unprocessable Entity` (FastAPI отклоняет запрос без обязательного заголовка) |
| 2 | Submit на несуществующий `task_id=99999` | `404 Not Found`, не 500 | `404 Not Found` |
| 3 | Пустая строка `""` как ответ на `FILL_BLANK` | `200 OK`, `is_correct: false` (не ошибка) | `200 OK`, `is_correct: false` |
| 4 | Строка `"not-an-int"` как ответ на `SINGLE_CHOICE` (ожидается int) | Не выбрасывать исключение, вернуть `is_correct: false` | `200 OK`, `is_correct: false` (перехвачено в `check_answer`) |
| 5 | Невалидный JSON `"{this is not valid json"` для `JSON_FIX` | `200 OK`, `is_correct: false`, понятная причина в `detail` | `200 OK`, `is_correct: false`, `detail` содержит `"invalid JSON: ..."` |
| 6 | SQL-инъекция в теле ответа (`'; DROP TABLE tasks; --`) | Ответ обрабатывается как обычная строка, таблица не затронута | `200 OK`, `is_correct: false`; последующий `GET /api/tasks/3` подтверждает, что таблица цела (используется SQLAlchemy ORM с параметризованными запросами — SQL-инъекция невозможна в принципе) |
| 7 | Очень длинная строка ответа (20 000 символов) | Не должно приводить к таймауту/ошибке | `200 OK`, `is_correct: false` |
| 8 | `null` как ответ на `MULTIPLE_CHOICE` | Не должно приводить к 500 | `200 OK`, `is_correct: false` |
| 9 | `MATCHING`-ответ с лишними непредусмотренными ключами (`{"99": 99}` вдобавок к верным парам) | Лишние ключи игнорируются, верные пары всё равно засчитываются | `200 OK`, `is_correct: true` (лишние ключи не участвуют в подсчёте `score`) |

## Ключевое архитектурное решение, обеспечившее устойчивость

`check_answer()` в `services/checker.py` оборачивает вызов каждой
стратегии проверки в `try/except Exception`, что гарантирует: любая
ошибка парсинга/сравнения (`ValueError`, `TypeError`, `KeyError`,
`json.JSONDecodeError` и т.п.) конвертируется в обычный отрицательный
результат проверки (`is_correct=False`), а не всплывает как
необработанное исключение FastAPI (которое привело бы к `500 Internal
Server Error` и раскрытию деталей реализации в ответе).
