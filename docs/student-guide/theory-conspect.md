# Основы REST и HTTP-методы

## Что такое REST

REST (Representational State Transfer) — архитектурный стиль для проектирования
сетевых API, предложенный Роем Филдингом в 2000 году в его диссертации. REST
API строится вокруг **ресурсов** — сущностей предметной области (пользователь,
заказ, товар), у каждого ресурса есть уникальный идентификатор (URI). Важно
понимать: REST — это не протокол и не стандарт, а набор архитектурных
ограничений (constraints); API, следующий не всем ограничениям, часто называют
"RESTful" или "REST-подобным", а не "чистым REST".

Ключевые принципы REST:

- **Клиент-сервер** — разделение ответственности между клиентом и сервером;
  они могут развиваться независимо, пока контракт API не меняется.
- **Отсутствие состояния (statelessness)** — каждый запрос содержит всю
  информацию, необходимую для его обработки (включая аутентификацию); сервер
  не хранит состояние сессии клиента между запросами. Это упрощает
  масштабирование: любой сервер из пула может обработать любой запрос.
- **Кэшируемость** — ответы должны явно указывать, можно ли их кэшировать
  (заголовки `Cache-Control`, `ETag`), и на какой срок.
- **Единообразный интерфейс** — ресурсы идентифицируются через URI,
  манипулируются через представления (JSON/XML), сообщения самоописательны
  (например, `Content-Type` явно указывает формат тела).
- **Слоистая система** — клиент не обязан знать, взаимодействует ли он
  напрямую с сервером или через промежуточные слои (прокси, балансировщики,
  CDN, API-шлюзы).

## HTTP-методы и их семантика

| Метод | Назначение | Safe | Idempotent | Тело запроса |
|---|---|---|---|---|
| GET | получить представление ресурса | да | да | обычно отсутствует |
| POST | создать ресурс / неидемпотентное действие | нет | нет | обычно есть |
| PUT | полностью заменить ресурс | нет | да | есть (полный ресурс) |
| PATCH | частично обновить ресурс | нет | нет* | есть (частичные поля) |
| DELETE | удалить ресурс | нет | да | обычно отсутствует |
| HEAD | как GET, но без тела ответа | да | да | отсутствует |
| OPTIONS | список поддерживаемых методов | да | да | отсутствует |

*PATCH формально не обязан быть идемпотентным, но многие практические
реализации (JSON Merge Patch, RFC 7396) делают его идемпотентным: повторное
применение одного и того же патча даёт тот же результат.

**Safe (безопасный) метод** не изменяет состояние сервера (только чтение).
Safe-методы можно свободно кэшировать, предзагружать (браузерный prefetch) и
повторять при сбоях сети без риска побочных эффектов.

**Idempotent (идемпотентный) метод** — повторное выполнение одного и того же
запроса приводит к тому же состоянию сервера, что и однократное выполнение.
Например, `DELETE /orders/5` идемпотентен: после первого удаления ресурс
исчезает, повторный DELETE ничего не меняет (хотя может вернуть 404 вместо
204 — код ответа может отличаться, но **состояние сервера** одинаково, и
именно это определение идемпотентности).

`POST` — единственный из основных методов, который *не* идемпотентен и *не*
safe: два одинаковых `POST /orders` обычно создают два разных заказа с разными
`id`. Именно поэтому браузеры предупреждают "Отправить форму повторно?" при
обновлении страницы после POST — это не выглядит проблемой на первый взгляд,
но повторная отправка формы оплаты создаст второй платёж.

## Пример полного цикла запрос-ответ

Разберём создание ресурса `PUT` vs `POST` на примере заказа.

**Создание нового заказа (POST — сервер сам генерирует id):**

```
POST /orders HTTP/1.1
Content-Type: application/json

{"user_id": 42, "items": [{"sku": "ABC-1", "qty": 2}]}
```

```
HTTP/1.1 201 Created
Location: /orders/1001
Content-Type: application/json

{"id": 1001, "user_id": 42, "status": "pending", "items": [...]}
```

**Полная замена существующего заказа (PUT — клиент указывает id и весь ресурс):**

```
PUT /orders/1001 HTTP/1.1
Content-Type: application/json

{"user_id": 42, "status": "cancelled", "items": [{"sku": "ABC-1", "qty": 2}]}
```

```
HTTP/1.1 200 OK
Content-Type: application/json

{"id": 1001, "user_id": 42, "status": "cancelled", "items": [...]}
```

Обратите внимание: в `PUT`-запросе клиент прислал **весь** ресурс целиком
(включая поля, которые не менялись, например `items`) — это ключевое отличие
от `PATCH`, где присылают только изменяемые поля:

```
PATCH /orders/1001 HTTP/1.1
Content-Type: application/json

{"status": "cancelled"}
```

## Частые ошибки проектирования

**Ошибка 1: GET с побочными эффектами.**
Использование `GET` для операций с побочными эффектами (например,
`GET /users/5/delete` вместо `DELETE /users/5`) нарушает контракт HTTP:
браузеры, прокси и краулеры могут вызывать GET-запросы автоматически
(prefetch, повторные попытки при сбоях сети), что приведёт к неожиданным
изменениям данных. Известный исторический случай: Google Web Accelerator
массово "нажимал" ссылки-GET для предзагрузки, что удаляло записи на сайтах,
использовавших GET-ссылки для удаления.

**Ошибка 2: PUT вместо PATCH для частичного обновления.**
Если клиент отправляет `PUT /orders/1001` только с полем `{"status": "cancelled"}`,
а сервер интерпретирует PUT как "замени весь ресурс этим телом" — все
остальные поля заказа (`items`, `user_id`) будут стёрты или обнулены. Это
частый источник багов при неверном выборе метода.

**Ошибка 3: POST для идемпотентных операций.**
Использование `POST /orders/1001/cancel` — нормально (команда-действие), но
если такая отмена реализована неидемпотентно (повторный вызов возвращает
ошибку "уже отменено" вместо того же успешного результата), это усложняет
retry-логику клиента при сетевых сбоях.

## Как самостоятельно написать HTTP-запрос

В практических заданиях этого блока нужно будет **написать запрос
самостоятельно**, а не выбрать вариант из списка. Формат простой: первая
строка — метод и путь, при необходимости на следующей строке — тело
запроса в формате JSON.

**Пример 1.** Получить список заказов пользователя с id 42:
```
GET /users/42/orders
```

**Пример 2.** Создать новый товар в каталоге:
```
POST /products
{"name": "Клавиатура", "price": 3990}
```

**Пример 3.** Частично обновить остаток товара с id 7 (только одно поле,
без замены всего ресурса):
```
PATCH /products/7
{"stock": 15}
```

Автопроверка сравнивает метод (без учёта регистра), путь (слэш в конце
не важен) и — если в задании требуется — тело запроса (сравнивается как
структура данных, а не как текст, поэтому порядок ключей и пробелы роли не
играют).

## Чек-лист перед отправкой запроса

- Соответствует ли выбранный метод характеру операции (чтение → GET,
  создание → POST, полная замена → PUT, частичное изменение → PATCH,
  удаление → DELETE)?
- Не изменяет ли запрос состояние сервера, если это GET?
- Не приведёт ли повторная отправка запроса (например, из-за повторного
  клика или сетевого сбоя) к нежелательному дублированию — если это POST
  без явной защиты идемпотентности?
- Указан ли `Content-Type: application/json`, если в запросе есть тело?

## Дополнительный пример: одна и та же сущность, разные методы

Разбор на сквозном примере ресурса `/articles/15`:

| Что нужно сделать | Запрос |
|---|---|
| Прочитать статью | `GET /articles/15` |
| Полностью переписать статью | `PUT /articles/15` (тело — весь объект статьи) |
| Исправить только заголовок | `PATCH /articles/15` (тело — `{"title": "..."}`) |
| Удалить статью | `DELETE /articles/15` |
| Опубликовать статью (не CRUD-действие) | `POST /articles/15/publish` |

Обратите внимание на последнюю строку: "опубликовать" — это не просто
изменение одного поля `status`, а бизнес-событие с побочными эффектами
(например, рассылка уведомлений подписчикам), поэтому многие API
намеренно выделяют такие действия в отдельный `POST`-эндпоинт с
существительным-командой в конце пути, а не прячут их за `PATCH`.

# Коды ответов HTTP и обработка ошибок

## Классы кодов состояния

- **1xx** — информационные (используются редко в REST API, например
  `100 Continue`).
- **2xx** — успех: `200 OK`, `201 Created`, `202 Accepted`, `204 No Content`.
- **3xx** — перенаправление: `301 Moved Permanently`, `304 Not Modified`.
- **4xx** — ошибка клиента: `400 Bad Request`, `401 Unauthorized`,
  `403 Forbidden`, `404 Not Found`, `405 Method Not Allowed`, `409 Conflict`,
  `422 Unprocessable Entity`, `429 Too Many Requests`.
- **5xx** — ошибка сервера: `500 Internal Server Error`, `502 Bad Gateway`,
  `503 Service Unavailable`.

Правило для клиента: код 4xx означает "переделай запрос" (повторная отправка
того же запроса без изменений бессмысленна), код 5xx означает "проблема на
сервере, можно попробовать повторить позже" (эти коды безопасно ретраить с
экспоненциальной задержкой).

## Наиболее важные коды для REST API — с примерами

**`200 OK`** — успешный GET/PUT/PATCH с телом ответа.

```
GET /orders/1001 HTTP/1.1
```
```
HTTP/1.1 200 OK
Content-Type: application/json

{"id": 1001, "status": "pending"}
```

**`201 Created`** — успешное создание ресурса через POST; обычно
сопровождается заголовком `Location` с URI нового ресурса (см. пример в
предыдущем блоке теории).

**`204 No Content`** — успех без тела ответа (например, DELETE):

```
DELETE /orders/1001 HTTP/1.1
```
```
HTTP/1.1 204 No Content
```

**`400 Bad Request`** — синтаксически некорректный запрос (например,
невалидный JSON или отсутствует обязательное поле):

```json
{"type": "https://api.example.com/errors/bad-request",
 "title": "Malformed JSON body",
 "status": 400}
```

**`401 Unauthorized`** — отсутствует или невалидна аутентификация (несмотря
на название, семантически это "не аутентифицирован", а не "нет прав" —
для отсутствия прав используется 403).

**`403 Forbidden`** — аутентификация есть, но прав недостаточно. Различие
401 vs 403 часто путают: 401 — "представьтесь", 403 — "мы знаем кто вы, но
вам сюда нельзя".

**`404 Not Found`** — ресурс с данным идентификатором не существует
(или — по соображениям безопасности — существует, но клиенту не положено
знать о его существовании; некоторые API намеренно возвращают 404 вместо
403, чтобы не раскрывать сам факт существования чужого ресурса).

**`409 Conflict`** — запрос конфликтует с текущим состоянием ресурса,
например, попытка создать пользователя с email, который уже занят:

```json
{"type": "https://api.example.com/errors/email-taken",
 "title": "Email already registered",
 "status": 409,
 "detail": "user@example.com is already associated with another account"}
```

**`422 Unprocessable Entity`** — запрос синтаксически корректен (валидный
JSON), но не прошёл бизнес-валидацию полей, например `{"age": -5}`.

**`429 Too Many Requests`** — превышен лимит запросов (подробнее в блоке про
rate limiting).

Частая ошибка — возвращать `200 OK` с телом `{"error": "..."}`. Это ломает
семантику HTTP: клиенты, кэши, мониторинг и автоматические retry-механизмы
ориентируются в первую очередь на код статуса, а не на разбор тела ответа.
Так, HTTP-кэш может закэшировать "ошибочный" `200 OK` как валидный успешный
ответ.

## Единый формат ошибок: RFC 7807 Problem Details

Чтобы клиенту не приходилось парсить разные форматы ошибок для разных
эндпоинтов одного и того же API, RFC 7807 определяет единую структуру тела
ошибки с media type `application/problem+json`:

```json
{
  "type": "https://example.com/errors/insufficient-funds",
  "title": "Недостаточно средств на счёте",
  "status": 409,
  "detail": "Баланс счёта 42 составляет 100 руб., требуется 150 руб.",
  "instance": "/accounts/42/withdraw"
}
```

Обязательные поля: `type` (идентификатор типа ошибки, часто URI — по нему
можно опубликовать документацию с описанием конкретной ошибки), `title`
(короткое человекочитаемое описание, одинаковое для всех ошибок этого
`type`), `status` (дублирует HTTP-код ответа для удобства логирования, если
тело ошибки логируется отдельно от заголовков). `detail` и `instance` —
опциональны, но крайне полезны для отладки конкретного случая: `detail`
объясняет именно эту ошибку (а не тип вообще), `instance` указывает, к
какому именно запросу относится ошибка.

## Как это делают реальные API

**GitHub API** использует свой формат (не RFC 7807, но похожий по духу),
включающий массив `errors` для валидационных ошибок сразу по нескольким
полям:

```json
{
  "message": "Validation Failed",
  "errors": [
    {"resource": "Issue", "field": "title", "code": "missing_field"}
  ],
  "documentation_url": "https://docs.github.com/rest/issues/issues#create-an-issue"
}
```

**Stripe API** использует другой подход: код ошибки, человекочитаемое
сообщение и указание конкретного невалидного параметра:

```json
{
  "error": {
    "type": "invalid_request_error",
    "message": "Amount must be at least 50 cents",
    "param": "amount"
  }
}
```

Оба примера показывают общий принцип, который важнее конкретного формата:
**единый, предсказуемый и документированный формат ошибки для всего API**,
а не произвольная структура на усмотрение каждого эндпоинта.

## Чек-лист выбора кода ответа

Простое дерево решений при проектировании эндпоинта:

1. Запрос обработан успешно?
   - Нет → см. пункт 2.
   - Да → вернуть тело в ответе? Да — `200 OK`; ресурс только что создан —
     `201 Created`; тела нет — `204 No Content`.
2. Проблема на стороне клиента (неверные данные, нет прав, ресурс не
   найден) или на стороне сервера?
   - Клиент прислал синтаксически некорректные данные → `400`.
   - Клиент не аутентифицирован → `401`.
   - Клиент аутентифицирован, но прав недостаточно → `403`.
   - Ресурса с таким id не существует → `404`.
   - Запрос корректен, но конфликтует с текущим состоянием (например,
     повторная регистрация того же email) → `409`.
   - Запрос корректен по синтаксису, но не проходит бизнес-валидацию
     (например, отрицательная цена) → `422`.
   - Сервер не смог обработать запрос из-за внутренней ошибки → `500`.

## Одна операция — разные коды в зависимости от ситуации

Частый вопрос новичков: "если я делаю `POST /subscriptions` дважды с одним
и тем же телом, что должен вернуть сервер?" Здесь нет единственно верного
ответа — зависит от бизнес-логики API:

- Если повторный `POST` должен создать **второй** независимый ресурс —
  сервер вернёт `201 Created` оба раза (два разных `id`).
- Если API реализует семантику "создать, если не существует" (upsert) —
  первый запрос вернёт `201 Created`, повторный с теми же данными вернёт
  `200 OK` с уже существующим ресурсом (без создания дубликата) — это
  осознанная альтернатива стандартной неидемпотентности `POST`, которую
  стоит явно документировать, так как она нарушает ожидания из блока 1.

## Ещё примеры форматов ошибок из индустрии

**Twilio API** включает числовой internal error code со ссылкой на страницу
документации по конкретной ошибке:

```json
{
  "code": 21211,
  "message": "The 'To' number is not a valid phone number.",
  "more_info": "https://www.twilio.com/docs/errors/21211",
  "status": 400
}
```

**PayPal API** — похожий на RFC 7807 подход, но с массивом `details` для
множественных ошибок валидации сразу:

```json
{
  "name": "VALIDATION_ERROR",
  "message": "Invalid request - see details",
  "details": [
    {"field": "/amount/value", "issue": "MISSING_REQUIRED_PARAMETER"}
  ]
}
```

# Именование ресурсов и версионирование API

## Конвенции именования URL

- Ресурсы именуются **существительными во множественном числе**: `/users`,
  `/orders`, а не `/getUsers` или `/user`.
- Действия над коллекцией и конкретным элементом:
  - `GET /orders` — список заказов
  - `POST /orders` — создать заказ
  - `GET /orders/{id}` — конкретный заказ
  - `PUT/PATCH /orders/{id}` — обновить
  - `DELETE /orders/{id}` — удалить
- **Вложенность ресурсов** отражает отношения: `GET /users/{id}/orders` —
  заказы конкретного пользователя. Вложенность больше 2 уровней
  (`/users/{id}/orders/{id}/items/{id}/...`) обычно сигнализирует о плохой
  декомпозиции ресурсов — вместо этого часто лучше дать вложенному ресурсу
  свой собственный корневой путь с фильтрацией: `GET /items?order_id={id}`.
- Глаголы в пути (`/getUsers`, `/createOrder`, `/orders/5/delete`) —
  анти-паттерн: HTTP-метод уже несёт в себе глагол-действие, дублирование
  избыточно и ломает единообразие интерфейса.
- Действия, которые плохо ложатся на CRUD (например, "отправить заказ в
  архив", "подтвердить оплату"), можно моделировать как под-ресурс/команду:
  `POST /orders/{id}/archive` — здесь `archive` существительное-команда, а
  не глагол в императиве.

### Таблица «плохо / хорошо»

| Плохо | Хорошо | Почему |
|---|---|---|
| `GET /getAllUsers` | `GET /users` | глагол избыточен, метод уже GET |
| `POST /user/create` | `POST /users` | ресурс всегда во множественном числе |
| `GET /orders/5/delete` | `DELETE /orders/5` | удаление — это метод, а не путь |
| `POST /orders/5/update-status` | `PATCH /orders/5` (тело `{"status": "..."}`) | обновление статуса — частный случай обновления ресурса |
| `GET /orders/5/cancel-order` | `POST /orders/5/cancel` | команда, не выражаемая через CRUD, но без лишнего глагола-дублёра |

## Версионирование API

Задача версионирования — дать возможность эволюционировать API, не ломая
существующих клиентов. Основные стратегии, с примерами из реальных API:

1. **Версия в URL**: `/api/v1/orders`, `/api/v2/orders`. Так делает, например,
   **Twitter/X API** (`api.twitter.com/2/tweets`). Просто, явно видно в
   логах и документации, но версия "протекает" в клиентский код — при
   миграции нужно менять base URL во всех местах.
2. **Версия в заголовке**: `Accept: application/vnd.myapp.v2+json` или
   кастомный `X-API-Version: 2`. Так делает, например, **GitHub API**
   (`Accept: application/vnd.github+json`) и **Stripe API**
   (заголовок `Stripe-Version: 2024-06-20`, где версия — дата релиза, а не
   номер). Чище с точки зрения REST (URI идентифицирует ресурс, а не его
   версию), но менее очевидно при отладке через браузер/curl — нужно помнить
   добавить заголовок.
3. **Версия как query-параметр**: `/orders?version=2` — редко рекомендуется,
   параметр легко случайно потерять при формировании ссылки, и он
   визуально смешивается с параметрами фильтрации.

Пример запроса с версией в заголовке (в стиле Stripe):

```
GET /v1/orders/1001 HTTP/1.1
Stripe-Version: 2024-06-20
Authorization: Bearer sk_live_...
```

## Эволюция API без breaking changes

Не каждое изменение требует новой версии. Правило: **добавление** нового
опционального поля в ответ/запрос — не breaking change; **удаление** или
**изменение типа/семантики** существующего поля — breaking change. Практика
"tolerant reader": клиенты должны игнорировать незнакомые поля в ответе, а не
падать на них.

**Пример не breaking change** (безопасно добавить, старые клиенты просто
игнорируют новое поле):

```diff
 {
   "id": 1001,
   "status": "pending",
+  "estimated_delivery": "2026-07-10"
 }
```

**Пример breaking change** (переименование поля `status` в `order_status`
сломает всех клиентов, которые читают `response.status`):

```diff
 {
   "id": 1001,
-  "status": "pending"
+  "order_status": "pending"
 }
```

Правильный способ переименовать поле без breaking change — временно
добавить **оба** поля (новое и старое, синхронизированные), объявить старое
`deprecated` в документации, дать клиентам время на миграцию, и только затем
(возможно, в следующей мажорной версии) удалить старое поле.

## Чек-лист именования ресурсов

- Ресурс называется существительным во множественном числе, без глаголов
  в пути?
- Вложенность не превышает 2 уровней?
- Одинаковые сущности называются одинаково во всём API (не `user` в одном
  месте и `customer` в другом для одного и того же понятия)?
- Идентификаторы в пути, а не в query-параметрах (`/orders/42`, а не
  `/orders?id=42`) — query-параметры зарезервированы для фильтрации,
  сортировки и пагинации, а не для идентификации конкретного ресурса?
- Не потребуется ли для этого изменения новая версия API, или его можно
  внести как дополнение, не ломающее текущих клиентов?

## Практика: как писать вложенные пути

Задания этого блока также используют формат "напишите запрос
самостоятельно" (см. подробности в блоке 1). Пример на вложенных
ресурсах — получить конкретный комментарий к конкретной статье:

```
GET /articles/15/comments/301
```

А вот пример **неправильного** пути, который вы могли бы написать по
ошибке (глагол в пути вместо метода):

```
GET /articles/15/getComments
```

— автопроверка засчитает такой путь как неверный, даже если сам сервер
гипотетически мог бы на него ответить: часть задания — потренироваться
именно в **конвенции**, а не только в том, "что технически сработает".

## Версионирование на практике: три реальных подхода рядом

| API | Стратегия | Пример |
|---|---|---|
| Twitter/X | версия в пути | `GET /2/tweets/12345` |
| GitHub | версия через `Accept`-заголовок | `Accept: application/vnd.github+json` |
| Stripe | версия через кастомный заголовок с датой | `Stripe-Version: 2024-06-20` |

Подход Stripe интересен тем, что версия — не номер (`v1`, `v2`), а дата
релиза: это позволяет выпускать множество мелких, независимо
документированных изменений без необходимости придумывать, когда именно
инкрементировать "большую" версию.

# Аутентификация и авторизация

## Authentication vs Authorization

- **Authentication (аутентификация, AuthN)** — подтверждение личности: "кто
  вы?". Результат — идентификатор субъекта (user id).
- **Authorization (авторизация, AuthZ)** — проверка прав: "что вам разрешено
  делать?". Выполняется после аутентификации, на основе ролей/скоупов.

Пример путаницы на практике: сервер вернул `403 Forbidden` на запрос без
токена вообще — это неверно, должно быть `401 Unauthorized` (клиент ещё не
аутентифицирован, серверу нечего проверять на предмет прав). `403` уместен
только когда сервер уже знает, кто делает запрос, и запрещает именно этому
известному субъекту.

## Основные механизмы аутентификации в API

**API key** — статичный секретный токен, передаётся в заголовке
(`X-API-Key`) или query-параметре. Просто, но нет встроенного механизма
истечения срока действия или ограничения прав.

```
GET /weather?city=Moscow HTTP/1.1
X-API-Key: 3f7a9c1e-...
```

**Basic Auth** — логин:пароль в заголовке `Authorization: Basic
base64(user:pass)`. Требует HTTPS, так как base64 — не шифрование, а просто
кодирование (обратимо за секунду).

```
GET /admin/reports HTTP/1.1
Authorization: Basic YWRtaW46c2VjcmV0MTIz
```

**OAuth 2.0** — протокол делегированной авторизации: приложение получает
временный access token от имени пользователя без доступа к его паролю.
Наиболее распространённый flow — **Authorization Code Flow**:

1. Приложение перенаправляет пользователя на сервер авторизации:
   ```
   GET https://auth.example.com/authorize
       ?client_id=abc123
       &redirect_uri=https://myapp.com/callback
       &scope=orders:read+orders:write
       &response_type=code
   ```
2. Пользователь логинится и подтверждает запрошенные права (consent screen).
3. Сервер авторизации перенаправляет пользователя обратно с одноразовым
   `authorization code`:
   ```
   GET https://myapp.com/callback?code=AQC8xyz...
   ```
4. Приложение обменивает `code` (+ `client_secret`) на `access_token`
   напрямую через серверный запрос (без участия браузера пользователя —
   это критично: `client_secret` никогда не должен попадать в браузер):
   ```
   POST https://auth.example.com/token
   Content-Type: application/x-www-form-urlencoded

   grant_type=authorization_code&code=AQC8xyz...&client_id=abc123&client_secret=...
   ```
   ```json
   {"access_token": "eyJhbGciOi...", "refresh_token": "8xLOxBt...", "expires_in": 3600}
   ```
5. Приложение использует `access_token` для запросов к API-ресурсам.

**JWT (JSON Web Token)** — самодостаточный токен вида
`header.payload.signature` (три части в base64url, разделённые точками).
Пример декодированного payload:

```json
{
  "sub": "user_42",
  "iat": 1719900000,
  "exp": 1719903600,
  "scope": "orders:read orders:write"
}
```

`payload` содержит **claims** — утверждения о субъекте (`sub` — идентификатор
субъекта, `exp` — время истечения, `iat` — время выпуска, кастомные claims
вроде `scope`). Сервер проверяет `signature`, не обращаясь к БД — отсюда
масштабируемость (любой из N серверов может проверить токен независимо), но
и сложность немедленного отзыва токена до истечения `exp` (нужен отдельный
механизм — например, короткий `exp` + список отозванных токенов, или
короткоживущий access token + долгоживущий refresh token).

## Скоупы (scopes)

Скоуп — гранула прав доступа, привязанная к токену (например `orders:read`,
`orders:write`). Позволяет выдавать токен с минимально необходимыми правами
(принцип **least privilege**) — приложению, которому нужно только
показывать список заказов, не стоит выдавать токен со скоупом на их
удаление, даже если само приложение написано без багов: скомпрометированный
токен такого приложения нанесёт меньше вреда.

## Заголовок передачи токена

Стандартный способ передать bearer-токен (JWT или OAuth2 access token) —
заголовок `Authorization: Bearer <token>`:

```
GET /orders HTTP/1.1
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

## Чек-лист безопасной аутентификации API

- Токен передаётся только по HTTPS (никогда не по обычному HTTP)?
- Access-токен короткоживущий (минуты/часы), а не бессрочный?
- Есть механизм обновления (`refresh_token`) без повторного ввода пароля?
- Токен выдаётся с минимально необходимым набором скоупов (least
  privilege), а не универсальным доступом "на всё"?
- Секрет приложения (`client_secret`) хранится только на сервере
  приложения и никогда не передаётся в браузер/мобильное приложение?

## PKCE — расширение OAuth2 для мобильных и SPA-приложений

Классический Authorization Code Flow предполагает, что `client_secret`
хранится в безопасном месте (на бэкенде приложения). Но у мобильных
приложений и SPA (одностраничных веб-приложений без выделенного бэкенда)
нет безопасного места для хранения секрета — весь код доступен
пользователю. Для таких клиентов используется **PKCE** (Proof Key for
Code Exchange, RFC 7636):

1. Приложение перед авторизацией генерирует случайную строку
   `code_verifier` и её хэш `code_challenge = SHA256(code_verifier)`.
2. `code_challenge` отправляется на шаге редиректа на сервер авторизации
   (вместо/вместе с `client_secret`).
3. При обмене `authorization code` на токен приложение присылает
   исходный `code_verifier`; сервер сам вычисляет его хэш и сверяет с
   ранее полученным `code_challenge`.

Это доказывает, что токен запрашивает то же приложение, которое
инициировало авторизацию, без необходимости хранить постоянный секрет.

## Скоупы на практике: как выглядит отказ в доступе

Если токен не содержит нужного скоупа, сервер должен вернуть `403
Forbidden` (не `401` — токен валиден, аутентификация прошла успешно,
просто прав недостаточно), например:

```json
{
  "type": "https://api.example.com/errors/insufficient-scope",
  "title": "Insufficient scope",
  "status": 403,
  "detail": "This action requires the 'orders:write' scope"
}
```

# Пагинация, фильтрация и идемпотентность запросов

## Пагинация

Возврат больших коллекций целиком — плохая практика: и сеть, и сервер, и
клиент тратят ресурсы на данные, которые пользователь может никогда не
увидеть (например, страницу 500 из 10000). Два основных подхода:

**Offset-based (страничная)**: `GET /orders?limit=20&offset=40`. Проста в
реализации (`LIMIT/OFFSET` в SQL), поддерживает переход на произвольную
страницу ("покажи страницу 7"), но:

- деградирует по производительности на больших offset — база данных всё
  равно должна просканировать и отбросить все `offset` пропускаемых строк;
- неустойчива к вставке/удалению строк между запросами страниц: если между
  запросом страницы 1 и страницы 2 кто-то добавил новую запись в начало
  списка, все последующие элементы "сдвинутся" — пользователь либо увидит
  дубликат, либо пропустит элемент.

**Cursor-based (курсорная)**: `GET /orders?limit=20&cursor=eyJpZCI6NDB9`.
Курсор — непрозрачный токен (обычно base64 от `{id последнего элемента,
дополнительные поля сортировки}`), кодирующий позицию последнего элемента
предыдущей страницы. Устойчива к изменениям данных между запросами
(`WHERE id > :cursor_id ORDER BY id LIMIT :limit` — новые записи в начале
списка не сдвигают уже выданные курсоры), эффективна на больших выборках, но
не позволяет прыгнуть на произвольную страницу — только "вперёд/назад"
последовательно.

### Как это делают реальные API

**GitHub API** использует cursor-подобную пагинацию через заголовок `Link`
(RFC 5988), а не через поле в теле ответа:

```
GET /users/octocat/repos?per_page=30 HTTP/1.1
```
```
HTTP/1.1 200 OK
Link: <https://api.github.com/user/repos?per_page=30&page=2>; rel="next",
      <https://api.github.com/user/repos?per_page=30&page=10>; rel="last"
```

**Stripe API** использует cursor-based пагинацию через параметры
`starting_after`/`ending_before` и поле `has_more` в ответе:

```json
{
  "object": "list",
  "data": [{"id": "ch_1", "amount": 2000}, {"id": "ch_2", "amount": 1500}],
  "has_more": true,
  "url": "/v1/charges"
}
```
Следующая страница запрашивается как
`GET /v1/charges?starting_after=ch_2`.

Универсальный вид ответа с курсорной пагинацией для собственного API:

```json
{
  "data": [],
  "pagination": {"next_cursor": "eyJpZCI6NjB9", "has_more": true}
}
```

## Фильтрация и сортировка

Query-параметры для фильтрации: `GET /orders?status=paid&created_after=2026-01-01`.
Сортировка: `GET /orders?sort=-created_at` (минус — по убыванию, как принято
в JSON:API и многих других соглашениях) или `GET /orders?sort=created_at&order=desc`
(два отдельных параметра — тоже валидный вариант). Важно **валидировать**
допустимые поля для фильтрации/сортировки на сервере через белый список —
иначе клиент сможет отсортировать выборку по служебным или закрытым полям
(например, по хэшу пароля) или спровоцировать дорогой запрос без индекса.

## Идемпотентность небезопасных запросов

`POST` не идемпотентен по умолчанию: повторная отправка одного и того же
запроса (например, из-за таймаута и автоматического retry на клиенте) может
создать дубликат — критично для операций вроде оплаты, где повторный запрос
из-за потерянного ответа сервера (запрос выполнился, но ответ не дошёл до
клиента) приведёт к двойному списанию денег.

Решение — заголовок **`Idempotency-Key`**: клиент генерирует уникальный
ключ (обычно UUID) для конкретной "попытки операции" и передаёт его в
каждом (в том числе повторном) запросе. Сервер запоминает результат первого
запроса с данным ключом (обычно на 24 часа) и при повторном запросе с тем же
ключом возвращает **сохранённый** результат, не выполняя операцию повторно.

```
POST /payments HTTP/1.1
Idempotency-Key: 5f2b1e3a-6f2c-4d1a-9e3b-1234567890ab
Content-Type: application/json

{"amount": 1500, "currency": "RUB"}
```

**Stripe API** реализует именно этот механизм под тем же именем заголовка
`Idempotency-Key` — если клиент повторит точно такой же POST-запрос с тем же
ключом (например, из-за таймаута соединения), Stripe вернёт результат первой
успешной попытки вместо создания второго платежа, причём даже с тем же
HTTP-кодом ответа, что был у первого запроса.

## Чек-лист корректной пагинации

- Есть ли ограничение по умолчанию на `limit`, если клиент его не указал
  (чтобы случайно не отдать миллион записей одним ответом)?
- Есть ли верхняя граница на `limit`, которую клиент не может превысить?
- Для курсорной пагинации — является ли курсор непрозрачным (клиент не
  должен полагаться на его внутреннюю структуру, она может измениться)?
- Возвращается ли признак "есть ли ещё данные" (`has_more` или наличие
  `next_cursor`), чтобы клиент не пытался угадывать по количеству
  элементов в ответе?

## Практика: запрос с пагинацией и фильтрами вместе

Пример запроса, комбинирующего фильтрацию, сортировку и пагинацию —
именно так может выглядеть ответ на задание "напишите запрос" из этого
блока:

```
GET /orders?status=paid&sort=-created_at&limit=20&cursor=eyJpZCI6NDB9
```

Разбор параметров: `status=paid` — фильтр по статусу, `sort=-created_at`
— сортировка по убыванию даты создания, `limit=20` — размер страницы,
`cursor=...` — курсор, полученный из предыдущего ответа API.

## Пагинация — это не то же самое, что rate limiting

Начинающие иногда путают эти два механизма, хотя они решают разные
задачи: **пагинация** ограничивает размер одного ответа (сколько записей
вернуть за раз), а **rate limiting** (см. следующий блок теории)
ограничивает частоту запросов во времени (сколько запросов можно сделать
за минуту/час). API может — и обычно должен — использовать оба механизма
одновременно и независимо друг от друга.

# Документация API, rate limiting и HATEOAS

## OpenAPI / Swagger

**OpenAPI Specification** (ранее Swagger) — машиночитаемый формат (YAML/JSON)
для описания REST API: эндпоинты, параметры, схемы тел запросов/ответов,
коды ответов, схемы аутентификации. По OpenAPI-документу можно автоматически
сгенерировать интерактивную документацию (**Swagger UI**), клиентские SDK,
серверные заглушки и тесты контракта.

Фрагмент OpenAPI-описания одного эндпоинта (упрощённо):

```yaml
paths:
  /orders/{id}:
    get:
      summary: Получить заказ по id
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: Заказ найден
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Order'
        '404':
          description: Заказ не найден
```

FastAPI (использованный в этом же проекте) — хороший живой пример: он
автоматически генерирует OpenAPI-схему из аннотаций типов Python и
Pydantic-моделей (вам не нужно писать YAML вручную — он выводится из кода), а
интерактивная документация доступна по умолчанию на `/docs` (Swagger UI) и
`/redoc` (ReDoc). Стоит открыть и изучить прямо в текущем проекте —
`http://localhost:8000/docs` — и сравнить её со списком эндпоинтов из блока
3 архитектурного отчёта данной практики.

Хорошая документация API включает: примеры запросов/ответов для каждого
эндпоинта (не только схему типов), описание кодов ошибок, changelog версий,
гайд по аутентификации ("Quickstart" за 5 минут для нового разработчика).

## Rate limiting (ограничение частоты запросов)

Защищает сервер от перегрузки и злоупотреблений — как умышленных (DDoS,
перебор паролей), так и случайных (клиент с багом в retry-логике,
опрашивающий API в цикле без задержки). Сервер отслеживает количество
запросов от клиента (по API-ключу/IP/user id) за окно времени и отклоняет
превышающие лимит запросы кодом `429 Too Many Requests`.

Информация о лимитах обычно передаётся в заголовках ответа (устоявшийся
де-факто стандарт, хотя официально не зафиксирован единым RFC — у разных
провайдеров названия заголовков немного отличаются):

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 42
X-RateLimit-Reset: 1717000000
Retry-After: 30
```

`Retry-After` особенно важен при ответе `429` — подсказывает клиенту, через
сколько секунд (или к какой дате) имеет смысл повторить запрос, вместо того
чтобы клиент угадывал задержку сам.

**GitHub API** — характерный реальный пример: лимит 5000 запросов в час для
аутентифицированных пользователей, отражается в заголовках каждого ответа:

```
GET /user/repos HTTP/1.1
Authorization: Bearer ghp_...
```
```
HTTP/1.1 200 OK
X-RateLimit-Limit: 5000
X-RateLimit-Remaining: 4987
X-RateLimit-Reset: 1717003600
```

## HATEOAS

**HATEOAS** (Hypermedia As The Engine Of Application State) — принцип из
оригинального REST Филдинга: ответ API должен содержать не только данные, но
и **ссылки на доступные из текущего состояния действия**, чтобы клиент не
хардкодил структуру URL заранее, а "перемещался" по API так же, как
пользователь по гипертексту (кликая по ссылкам HTML-страницы, а не вводя
адреса вручную).

```json
{
  "id": 42,
  "status": "pending",
  "amount": 1500,
  "_links": {
    "self": {"href": "/orders/42"},
    "cancel": {"href": "/orders/42/cancel", "method": "POST"},
    "pay": {"href": "/orders/42/pay", "method": "POST"}
  }
}
```

Ключевая идея — **ссылки отражают текущее состояние**: если тот же заказ уже
оплачен, ответ не должен содержать ссылку `pay` (действие недоступно), но
должен появиться `refund`:

```json
{
  "id": 42,
  "status": "paid",
  "amount": 1500,
  "_links": {
    "self": {"href": "/orders/42"},
    "refund": {"href": "/orders/42/refund", "method": "POST"}
  }
}
```

Это избавляет клиента от необходимости самому знать бизнес-правила ("после
оплаты нельзя платить повторно, но можно вернуть деньги") — он просто
показывает пользователю только те кнопки, ссылки на которые реально пришли
в ответе.

На практике полноценный HATEOAS в вебе встречается редко (клиенты чаще
жёстко знают структуру API из документации, что проще в разработке), но
даже частичное применение — как в примере выше — заметно упрощает
клиентскую логику и делает API самоописательным.

## Чек-лист хорошей документации API

- Для каждого эндпоинта есть пример запроса и пример успешного ответа?
- Описаны все возможные коды ошибок для эндпоинта, а не только "успех"?
- Есть быстрый гайд по аутентификации ("получить токен за 5 минут")?
- Документация генерируется из кода (как OpenAPI-схема в FastAPI), а не
  поддерживается вручную отдельным файлом — иначе она гарантированно
  устареет и разойдётся с реальным поведением API?
- Есть changelog версий с датами и описанием breaking/non-breaking
  изменений?

## Sunset и Deprecation — как сообщить об отключении эндпоинта

Когда эндпоинт или версия API выводится из эксплуатации, стандарт RFC
8594 определяет заголовок `Sunset`, сообщающий дату отключения:

```
HTTP/1.1 200 OK
Deprecation: true
Sunset: Sat, 31 Dec 2026 23:59:59 GMT
Link: <https://docs.example.com/migration-v2>; rel="sunset"
```

Клиент (или его разработчик, увидевший это в логах) заранее узнаёт о
предстоящем отключении и успевает мигрировать — это продолжение идеи
"эволюция API без breaking changes" из блока 3, только на уровне
предупреждения о **будущем** breaking change, а не самого изменения.

## Webhooks: API "наоборот" и подпись запроса

Отдельная смежная тема — **webhooks**, когда не клиент опрашивает сервер,
а сервер сам присылает POST-запрос клиенту при наступлении события
(например, "платёж прошёл успешно"). Поскольку webhook-эндпоинт клиента
технически открыт в интернет, важна проверка подлинности: сервер
подписывает тело запроса секретным ключом и передаёт подпись в заголовке
(`Stripe-Signature`, `X-Hub-Signature-256` и т.п.), а клиент проверяет
подпись перед тем, как доверять содержимому запроса.
